01 — The Problem

Why are distributed
transactions hard?

In a monolith, a database transaction gives you ACID guarantees — Atomicity, Consistency, Isolation, Durability — for free. Microservices break this: each service owns its own database, so a business operation that spans services has no single transaction boundary.

⚡
Partial Failures
Service A succeeds, Service B crashes halfway. Data is inconsistent with no automatic rollback.
🌐
Network Partitions
The CAP theorem forces a choice between Consistency and Availability during a partition.
⏱
Latency Amplification
Cross-service locks compound latencies. Locking 3 services triples your worst-case wait time.
📐
No Single Truth
Each service's DB has its own view. Achieving global consistency requires coordination protocols.

02 — Pattern 1

Two-Phase Commit
2PC

The classic distributed transaction protocol. A Coordinator drives a two-step protocol across all participating services: Prepare then Commit.

How it works
Phase 1: Coordinator asks all participants to prepare and lock resources. Phase 2: If all vote YES → COMMIT. If any votes NO → ROLLBACK all.
Pros
ACID guarantees
Simple mental model
Strong consistency
Cons
Blocking locks
Single point of failure (coordinator)
High latency
Poor availability
When to use
Financial systems, legacy DB integrations, or scenarios where strict consistency outweighs availability. Rarely used in modern cloud-native systems.

03 — Pattern 2

The Saga Pattern

A Saga breaks a long business transaction into a sequence of local transactions, each publishing events or messages. If any step fails, compensating transactions are triggered to undo previous steps — no distributed lock required.

Key insight: Sagas achieve eventual consistency through compensating transactions rather than rollbacks. The system must be designed so compensations are always possible (idempotent and retryable).

✅
Pros
No distributed lock High availability Services stay loosely coupled Works across polyglot DBs
⚠️
Cons
Eventually consistent only Complex compensation logic Hard to debug Cyclic dependency risk

Sagas come in two flavours — Choreography and Orchestration — covered in depth next.

04 — Saga Variant A

Choreography
Event-Driven

No central coordinator. Each service listens for events on a message broker (Kafka, RabbitMQ) and reacts by performing its local transaction and publishing the next event.

📡
Core mechanics
Services publish domain events. Other services subscribe and react. The workflow emerges from the interactions — no single service "owns" the saga.
✅
Strengths
Truly decoupled Scales independently No single point of failure
⚠️
Challenges
Hard to visualize flow Event schema coupling Distributed tracing needed Risk of cyclic events
🏢
Best for
Simple, linear workflows; teams that value independence; systems where each service team owns its full lifecycle. Under ~4–5 steps.

05 — Saga Variant B

Orchestration
Command-Driven

A dedicated Saga Orchestrator (or Process Manager) sits in the middle. It sends commands to services and waits for responses. It holds the full state of the workflow.

🎯
Core mechanics
Orchestrator sends explicit commands; services reply with results. The orchestrator decides the next step, handles errors, and triggers compensations centrally.
✅
Strengths
Clear, visible workflow Centralized error handling Easy to audit & debug Handles complex branching
⚠️
Challenges
Orchestrator is a bottleneck Can become god object Couples orchestrator to all services
🏢
Best for
Complex, multi-step workflows with branching/retry logic. Regulated industries (fintech, healthcare) needing audit trails. Workflows with 5+ steps.

07 — Real World

What Top Tech
Companies Use

Uber
Uses Orchestration via their in-house Cadence workflow engine (now open-sourced as Temporal). Trip lifecycle — rider match, driver dispatch, payment, receipt — is a saga orchestrated as a durable state machine.
Temporal/Cadence Orchestration
Netflix
Heavy Choreography via Kafka. Each domain service reacts to events (stream started, billing event, encoding complete). They prioritize availability over strict consistency.
Kafka Choreography
Amazon
Uses both. Order fulfillment uses saga choreography (events between warehouse, payment, shipping). Internal workflows use AWS Step Functions (orchestration).
SNS/SQS Step Functions Hybrid
Stripe
Payment workflows use Orchestration with idempotency keys. Complex payment retry logic, dispute handling, and payout flows need explicit state management and auditability.
Orchestration Idempotent
LinkedIn
Built on Kafka — the company open-sourced it. Feed updates, connection events, and notifications all flow as choreographed sagas through event streams.
Kafka Choreography
Airbnb
Booking flow (search → hold → payment → confirm → notify host) uses a saga orchestration with compensation logic for each failure point.
Orchestration Saga

🏆 The Industry Consensus (2024–2025)

Saga Orchestration via Temporal / AWS Step Functions is the dominant choice for new systems handling complex business workflows. It offers the best tradeoff: observable state, manageable compensations, good scalability, and battle-tested tooling.

Choreography wins for high-throughput, decoupled domains (event pipelines, analytics, notifications) where the workflow is simple and teams want full autonomy.

2PC is effectively legacy for microservices — only justified when integrating with older RDBMS systems that mandate XA transactions.

Orchestration (Temporal) for complex flows   Choreography (Kafka) for event pipelines   2PC only for legacy
